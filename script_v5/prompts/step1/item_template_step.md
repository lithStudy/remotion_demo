## 🎯 选型决策流程

对**每个待输出的 item**，分析text内容，寻找适合表达的画面模板，按以下顺序**自上向下**查找，**命中第一条即停止**。

### 决策树

**Step 1 — 数据型**（`narrativeRole` 为 data，或 `text` 含明确数值/百分比/倍率且强调量化结论）
- 两项指标同框对比（涨跌、前后、A/B）→ `STAT_COMPARE`（须能在口播中合理抽象出两个非负整数含义；若抽象勉强则勿选）
- 单指标完成度/占比结论 → `PROGRESS_RING`
- 单指标大字报（一个核心数）→ `KPI_HERO`

**Step 2 — 对错 / 双方案对立**
- 明确「别这样 vs 要这样」「错误做法 vs 正确做法」→ `DOS_AND_DONTS`
- **双方行为/立场对照**（你方 vs 他方/对方；讲道理 vs 甩链接；分号「；」两侧描述**对立**两方而非单纯并列举例）→ `SPLIT_COMPARE`  
  - 与并列多分点区分：**对仗用双分屏**；**同一论证下多个平级例子**用合并 + `LIST_MULTI_GROUP`，不要强行 `SPLIT_COMPARE`。

**Step 3 — 结构化多分点 / 清单 / 节拍**
- **2～4（至多 5）个可独立成画的并列主体**，且**当前 item 的 `text` 里真实展开了多个分点**（非仅总起句）→ `LIST_MULTI_GROUP`  
- **短步骤 / 强清单**（第一步…第二步… 或多条极短分点），无配图诉求 → `STEP_LIST`（**单条仅一个分点**不要用；**「标题 + 长解释」**不要用，见下）
- **一个方法/提醒/观点标题 + 随后 2～4 句解释展开**（同一 item 内）→ `METHOD_STACK`
- **同一论证、同一镜头内 2～4 段口播节拍递进**（如问→驳→收束；图随分段切换）→ `BEAT_SEQUENCE`

**Step 4 — 引用 / 术语 / 揭秘**
- 名言、著作、研究引用 → `QUOTE_CITATION`
- 解释专业术语/模型名 → `CONCEPT_CARD`
- **揭秘、拆穿、强调真相/底层逻辑**，且你能预期后续可为分段 `content` 配置 **至少 1 个高价值锚点短语** → `MAGNIFYING_GLASS`；**拿不准则不要选**（后续参数阶段需非空 `anchors`）  
  - **必要条件（缺一不可）**：该 item 内必须存在 1～2 句“揭示性落点句”（如「真正的 X 是 Y」「你以为…其实…」「底层逻辑是…」），并且其余口播主要是在围绕这个落点解释/展开。  
  - **禁止打包**：不要把“多个机制 + 多个例子 + 总结金句”全部塞进同一个 `MAGNIFYING_GLASS`（这类更适合 `BEAT_SEQUENCE` 分节拍，或拆成多个 item）。  
  - **与合并联动**：若你打算合并相邻条目来做 `MAGNIFYING_GLASS`，除非合并后仍满足合并硬限制（尤其 **≤ 60 字**）且仍只有一个清晰落点句，否则禁止为此合并。

**Step 5 — 高情绪 / 金句**
- 「不是…而是…」强认知翻转 → `COGNITIVE_SHIFT`
- 全片级金句、零图大字冲击 → `TEXT_FOCUS`（`narrativeRole=hook` **单独不足以**选本模板，须语气上真是短促强结论）

**Step 6 — 对话 / 发问 / 心声**
- 仅在满足以下 **强信号** 时才选 `CHAT_BUBBLE`（任一命中即可）：  
  - **显式对话格式**：出现 A：B：、多句互相回应、或“我：…/你：…”等对话回合；  
  - **显式弹幕/评论体**：出现“弹幕：/网友：/评论：/有人说：”等角色化心声（而非普通叙述）。  
- **反例（不要误选）**：仅有“你是不是也…”这类单句发问，后面紧跟场景举例/陈述来做开场共鸣时，通常应按 HOOK 处理，优先 `CENTER_FOCUS`（除非确实写成对话回合或弹幕体）。

**Step 7 — 时间轴**
- 有明确时间先后/阶段演进（去年→今年→未来 等）→ `TIMELINE`（2～3 个时间点/阶段）

**Step 8 — 默认**
- 平缓叙述、单图即可、无上述结构 → `CENTER_FOCUS`

## 🚨 选型避坑与自检清单（输出 JSON 前须过一遍）

1. **`CENTER_FOCUS` 禁令**：禁止「能合并却拆碎」后全选 `CENTER_FOCUS`。同一 scene 内**连续 2 个及以上** `CENTER_FOCUS`
2. **全场景 `CENTER_FOCUS` 比例**：若超过约 **60%**，视为异常，须重新检查是否遗漏结构化语义。
3. **`LIST_MULTI_GROUP`**：口播里**真实并列分点/并列举例**的数量须与后续 param 的 **`groups` 条数**一致（**2～5 组**，且不超过 schema `maxItems`）。**禁止**把同一句内的多处逗号停顿误当成多个并列主体；**禁止**按「一行字幕 = 一组图」设想（`content` 行数通常多于组数）。并列举例应按**例子/分点**聚合一组起始 `textIndex`，且规划上须**覆盖到该 item 口播最后一句**。总起句+后续分点应依赖 **Item 合并** 后再选。
4. **`STEP_LIST` 滥用**：单分点、「标题+长解释」须用 `METHOD_STACK` 或改分镜合并；全 scene 单分点 `STEP_LIST` 过多须重审 Step 3。
5. **`SPLIT_COMPARE` vs 并列举例**：**并列恐慌/焦虑类例子**（如坐飞机、刷短视频）不是双方对立的 `SPLIT_COMPARE`；通常优先 `CENTER_FOCUS`（开场共鸣）或在确需并列展示时选 `LIST_MULTI_GROUP`。
6. **`MAGNIFYING_GLASS`**：非揭秘句勿选；选则须在 `reasoning` 中说明预期锚点从句中哪类信息提取，并且必须存在清晰的“揭示性落点句”；禁止用它打包多个机制/多个例子。
7. **相邻 `BEAT_SEQUENCE`**：考虑在选型阶段合并为一个 item（仍需满足合并硬限制，尤其字数 ≤ 60）。
