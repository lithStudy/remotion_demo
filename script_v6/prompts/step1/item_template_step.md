## 🎯 选型决策流程

对**每个待输出的 item**，分析text内容，寻找适合表达的画面模板，按以下顺序**自上向下**查找，**命中第一条即停止**。

### 决策树

**Step 1 — 数据型**（`narrativeRole` 为 data，或 `text` 含明确数值/百分比/倍率且强调量化结论）
- 两项指标同框对比（涨跌、前后、A/B）→ `STAT_COMPARE`（须能在口播中合理抽象出两个非负整数含义；若抽象勉强则勿选）
- 单指标完成度/占比结论 → `PROGRESS_RING`
- 单指标大字报（一个核心数）→ `KPI_HERO`

**Step 2 — 对错 / 双方案对立**
- 明确「别这样 vs 要这样」「错误做法 vs 正确做法」→ `DOS_AND_DONTS`
- **双方行为/立场对照**（你方 vs 他方/对方；讲道理 vs 甩链接；分号「；」两侧描述**对立**两方而非单纯并列举例）→ `SPLIT_COMPARE`  
  - 与并列多分点区分：**对仗用双分屏**；**同一论证下多个平级例子**用合并 + `PANEL_GRID`，不要强行 `SPLIT_COMPARE`。

**Step 3 — 结构化多分点 / 清单 / 节拍**
- **2～4 个可独立成画的并列主题**（宫格，每块一图），且**当前 item 的 `text` 里真实展开了多个分点**（非仅总起句）→ `PANEL_GRID`（`panels` 条数须与真实并列分点一致，且不超过 schema `maxItems`）
- **短步骤 / 强清单**（第一步…第二步… 或多条极短分点），无配图诉求 → `STEP_LIST`（**单条仅一个分点**不要用；**「标题 + 长解释」**不要用，见下）
- **一个标题式核心（方法/建议/观点/卖点/亮点/结论） + 随后 2～4 句解释展开**（同一 item 内）→ `METHOD_STACK`
- **同一论证、同一镜头内 2～4 段口播节拍递进**（如问→驳→收束；图随分段切换）→ `BEAT_SEQUENCE`
- **因果链 / 机制传导**（因为…所以…导致…；同一 item 内 2～4 环，每环对应一句口播且**各环一图**，**非**年代/时间刻度演进）→ `CAUSE_CHAIN`（与 `TIMELINE` 区分：无「去年/阶段/年代」时间轴语义；与 `METHOD_STACK` 区分：非「一个标题 + 多句解释」）

**Step 4 — 引用 / 术语 / 聚焦强调**
- 名言、著作、研究引用 → `QUOTE_CITATION`
- 首次抛出需要“命名/标签化”的术语、概念、产品名、品牌关键词 → `CONCEPT_CARD`
- **需要把观众注意力“锁定在一个关键词/短语/关键细节/核心发现”上**，且你能预期后续可为分段 `content` 配置 **至少 1 个高价值锚点短语** → `MAGNIFYING_GLASS`；**拿不准则不要选**（后续参数阶段需非空 `anchors`）  
  - **必要条件（缺一不可）**：该 item 内必须存在 1～2 句“落点句”（如「关键在于 X」「核心卖点是 X」「真正重要的是 X」「结论是 X」「这一点决定了…」），并且其余口播主要是在围绕这个落点解释/展开。  
  - **禁止打包**：不要把“多个机制 + 多个例子 + 多个结论”全部塞进同一个 `MAGNIFYING_GLASS`（这类更适合 `BEAT_SEQUENCE` 分节拍，或拆成多个 item）。  
  - **与合并联动**：若你打算合并相邻条目来做 `MAGNIFYING_GLASS`，除非合并后仍满足合并硬限制（尤其 **≤ 60 字**）且仍只有一个清晰落点，否则禁止为此合并。

**Step 5 — 高情绪 / 金句**
- 「不是…而是…」强对立翻转（纠偏、观点翻转、定位重述均可）→ `COGNITIVE_SHIFT`
- 全片级金句、零图大字冲击 → `TEXT_FOCUS`（`narrativeRole=hook` **单独不足以**选本模板，须语气上真是短促强结论）；参数阶段大屏主文案用非空 `coreSentence`（string[]，每元素一行）；大屏高亮用 `coreSentenceAnchors`（每项 `coreSentenceAnchor` + 可选 `color`，子串须出现在各段 `coreSentence` 按顺序拼接后的全文内），**不要**填 `anchors` / `showFrom`

**Step 6 — 对话 / 发问 / 心声**
- 仅在满足以下 **强信号** 时才选 `CHAT_BUBBLE`（任一命中即可）：  
  - **显式对话格式**：出现 A：B：、多句互相回应、或“我：…/你：…”等对话回合；  
  - **显式弹幕/评论体**：出现“弹幕：/网友：/评论：/有人说：”等角色化心声（而非普通叙述）。  
- **反例（不要误选）**：仅有“你是不是也…”这类单句发问，后面紧跟场景举例/陈述来做开场共鸣时，通常应按 HOOK 处理，优先 `CENTER_FOCUS`（除非确实写成对话回合或弹幕体）。

**Step 7 — 时间轴**
- 有明确时间先后/阶段演进（去年→今年→未来 等）→ `TIMELINE`（2～3 个时间点/阶段）

**Step 7.5 — 收束清单**
- **收束段、行动检查项、要点复诵**：口播逐条对应清单行，需要**逐项显现 + 打勾**的闭环感，且**非**「第一/第二」编号短步骤体、**非**并列大图组 → `CHECKLIST_REVEAL`（与 `STEP_LIST` 区分：后者是步骤序号清单且无打勾语义；勿为用模板而碎镜）

**Step 8 — 默认**
- 平缓叙述、单图即可、无上述结构 → `CENTER_FOCUS`

## 🚨 选型避坑与自检清单（输出 JSON 前须过一遍）

1. **`CENTER_FOCUS` 禁令**：禁止「能合并却拆碎」后全选 `CENTER_FOCUS`。同一 scene 内**连续 2 个及以上** `CENTER_FOCUS`
2. **全场景 `CENTER_FOCUS` 比例**：若超过约 **60%**，视为异常，须重新检查是否遗漏结构化语义。
3. **`PANEL_GRID`**：口播里**真实并列分点/并列举例**的数量须与后续 param 的 **`panels` 条数**一致（**2～4 项**，且不超过 schema `maxItems`）。**禁止**把同一句内的多处逗号停顿误当成多个并列主体；**禁止**按「一行字幕 = 一宫格」设想（`content` 行数通常多于 `panels`）。每项 `showFrom` 须指向该分点对应的 **content 下标**，且规划上须**覆盖到该 item 口播最后一句**。总起句+后续分点应依赖 **Item 合并** 后再选。
4. **`STEP_LIST` 滥用**：单分点、「标题+长解释」须用 `METHOD_STACK` 或改分镜合并；全 scene 单分点 `STEP_LIST` 过多须重审 Step 3。
5. **`SPLIT_COMPARE` vs 并列举例**：把“同一观点下的多个平级例子/场景罗列”（如通勤/开会/健身等）误当成双方对立，是最常见误选；这类通常优先 `CENTER_FOCUS`（开场共鸣）或在确需并列展示时选 `PANEL_GRID`，不要强行 `SPLIT_COMPARE`。
6. **`MAGNIFYING_GLASS`**：没有明确“落点句/关键词/关键细节”就不要选；选则须在 `reasoning` 中说明锚点从哪句/哪类信息提取，并且必须存在清晰落点；禁止用它打包多个机制/多个例子/多个结论。
7. **相邻 `BEAT_SEQUENCE`**：考虑在选型阶段合并为一个 item（仍需满足合并硬限制，尤其字数 ≤ 60）。
